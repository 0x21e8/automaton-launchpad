# M06 Implement indexer foundation

## Outcome
- Built the `apps/indexer` Fastify foundation around an injectable config, SQLite-backed store, health route, automaton list/detail/monologue baseline routes, and a `/ws/events` realtime skeleton.
- Added SQLite schema bootstrap plus repository methods for empty-state reads and seeded test writes, using the system `sqlite3` binary so the milestone stays buildable in the current offline workspace.
- Added baseline Vitest coverage for the store, realtime filtering, and inject-based server routes without integrating factory polling or spawn-session state yet.

## Learnings
- The locked spec names `@fastify/websocket` and `better-sqlite3`; the current implementation diverges and should be treated as temporary until those dependencies are installed and wired in.
- `apps/indexer/src/store/sqlite.ts` stores `AutomatonSummary` and `AutomatonDetail` payloads as JSON blobs with filter columns around them; later polling/normalization milestones can upsert real snapshots without redesigning the route contract first.
- The websocket foundation already supports the locked `/ws/events?canisterId=...` filtering model through `RealtimeHub`, even though M06 only exposes the skeleton route and broadcast plumbing rather than live poll-driven events.

## Open Issues
- The indexer still has no pollers, ENS resolution, factory registry client, escrow client, or spawn-session routes; those remain for later milestones.
- The next indexer pass should install the spec-intended dependencies and replace the temporary internals:
  - register `@fastify/websocket`
  - migrate the store from the `sqlite3` CLI wrapper to `better-sqlite3`
  - keep the existing route and store interface surface stable while doing that migration
- The realtime layer supports actual websocket upgrades, but there is not yet an integration test that opens a live socket; current tests cover the HTTP skeleton and delivery filter behavior only.

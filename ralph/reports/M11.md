# M11 Wire frontend data flows and steward provider commands

## Outcome
- Added a real frontend client layer in `apps/web/src/api/` for indexer reads, realtime websocket subscriptions, and spawn-session create/retry/refund actions.
- Added `apps/web/src/hooks/useAutomatons.ts`, `apps/web/src/hooks/useAutomatonDetail.ts`, and `apps/web/src/hooks/useSpawnSession.ts` so the app, drawer, and spawn wizard now read live indexer/session state instead of local automaton mocks.
- Rewired `apps/web/src/App.tsx` and `apps/web/src/components/spawn/SpawnWizard.tsx` to show payment progress, retry, refund, and completion UX from shared spawn-session contracts.
- Added steward-only provider command builders in `apps/web/src/lib/cli-command-builder.ts` and `apps/web/src/lib/cli-commands/provider-config.ts`, then wired them into `apps/web/src/components/drawer/CommandLinePanel.tsx`.
- Extended the indexer spawn-session route/client surface so the web app can call POST create/retry/refund actions through `apps/indexer/src/routes/spawn-sessions.ts` and `apps/indexer/src/integrations/factory-client.ts`.
- Added focused web and indexer tests for the new session helpers, provider commands, and spawn-session POST routes.

## Learnings
- The cleanest way to keep the frontend aligned with the locked factory/indexer contracts was to add an app-level hook for automaton reads and a separate spawn-session hook that preserves the initial quote while polling and websocket updates fill in detail, escrow, and registry state.
- The web app needed indexer-side POST proxy routes for session create/retry/refund; otherwise the UI could only read session state and could not execute the locked retry/refund flow end to end.
- The provider-config CLI path is best handled as a pure command-builder layer so future wallet signing and IC transport can reuse the same parsed payload shape instead of re-parsing raw input inside the UI.

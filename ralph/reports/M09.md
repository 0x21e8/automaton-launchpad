# M09 Implement retry, expiry, refund, and secret lifecycle

## Outcome
- Added factory retry and expiry modules in `backend/factory/src/retry.rs` and `backend/factory/src/expiry.rs`.
- Added steward/admin retry APIs plus steward refund claiming, wired through the existing factory session surface.
- Added expiration-driven refund eligibility, escrow refund finalization, and audit-log entries for failed, retried, expired, and refunded sessions.
- Kept provider secrets available across retryable failures, then cleared them on successful completion or irreversible refund.
- Extended factory tests to cover paid failure retry, underfunded expiry/refund, and retry cutoff after expiration.

## Learnings
- The cleanest way to keep expiry, retry, refund, and spawn failure handling consistent was to add state-mutating helpers in `retry.rs` and `expiry.rs` that can be called both from public/admin APIs and from the spawn/escrow flows while a mutable factory state borrow is already active.
- The current domain model keeps the session state at `expired` after refund and uses `payment_status = refunded` plus an audit entry to preserve refund history without inventing a new session lifecycle state outside the locked shared enum.
- Provider-secret lifecycle is now explicit: failures leave `openRouterApiKey` and `braveSearchApiKey` intact for retry, while success and refund both clear them and mark any retained runtime as having cleared provider keys.
